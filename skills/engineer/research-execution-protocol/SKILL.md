# Research Execution Protocol

用于高不确定性、易反复失败、需要多轮验证的工程任务。

目标不是“更凶”，而是更稳：
- 减少空想式回答
- 减少未验证即宣称完成
- 减少卡住后立即向用户转嫁排障成本
- 提高任务闭环率与结果可复现性

## 触发条件

当满足任一条件时启用：

- 用户要求你修 bug、改代码、补功能、排查配置、修环境
- 任务涉及多文件修改、外部依赖、工具调用或实验验证
- 第一次方案失败，且仍存在可继续验证的路径
- 任务需要“定位原因 -> 修改 -> 验证 -> 复盘”完整闭环
- 用户明确表达：继续查、继续修、别停在分析

不适用于：
- 纯闲聊
- 低成本一次性问答
- 只需给概念解释、无需执行验证的场景

## 核心原则

### 1. 证据先于判断
先收集最小必要证据，再下结论。
不要在没有日志、代码位置、配置状态、返回结果的情况下断言根因。

### 2. 验证先于完成
除非已有可见证据，否则不要说“已经修好”“应该可以”“完成了”。
只有在满足至少一项时，才可宣称完成：
- 测试通过
- 命令成功执行
- 输出结果与目标一致
- 用户提供的失败路径被复现后已消失

### 3. 失败后不重复原思路
一次失败后，必须显式切换策略之一：
- 换根因假设
- 换检查位置
- 换工具路径
- 缩小问题规模
- 先做最小可复现
- 先绕过再回补

### 4. 局部改动优先
优先做最小、可验证、可回滚的改动。
避免在根因未明时大面积重构。

### 5. 显式区分事实与推断
输出中区分：
- `[FACT]` 已观察到的事实
- `[HYP]` 当前假设
- `[TEST]` 为验证假设采取的动作
- `[RESULT]` 动作结果
- `[NEXT]` 下一步

### 6. 不把压力转嫁给用户
能自己做的检查先自己做。
只有在以下情况才向用户索取额外信息：
- 缺失关键输入且无法从现有上下文/文件/工具中获得
- 外部权限不足
- 所有合理备用路径已试过仍失败

## 标准流程

### Step 1. 定义目标与完成标准
先在内部明确：
- 目标是什么
- 什么算完成
- 什么证据能证明完成

若目标模糊，优先从用户原话中抽取最保守目标，不扩写需求。

### Step 2. 建立最小问题模型
快速回答四件事：
1. 失败点在哪一层：输入 / 路由 / 代码 / 配置 / 依赖 / 环境 / 外部接口
2. 可否复现
3. 哪个证据最便宜
4. 哪种验证能最快排除一大片错误假设

### Step 3. 先做低成本高信息量检查
优先顺序：
1. 读相关代码 / 配置 / 文档
2. 看错误信息 / 日志 / 调用返回
3. 复现最小路径
4. 做单点修改或单点验证
5. 再考虑扩大改动面

### Step 4. 维护“假设账本”
每轮最多保留 1-3 个主假设。
每个假设都要绑定：
- 触发它的证据
- 如何验证
- 如何证伪

一旦被证伪，立即丢弃，不重复包装旧假设。

### Step 5. 改动后立即验证
任何改动之后，至少做一项贴近目标的验证：
- 单元测试
- 命令运行
- 静态检查
- 样例输入回放
- 关键路径 smoke test

### Step 6. 失败时执行升级策略
若当前路径失败，不要立刻结束。按顺序选择一个升级动作：
1. 缩小范围：最小可复现
2. 换假设：不要继续押注同一解释
3. 换观测：增加日志/打印/检查中间态
4. 换路径：从代码修复切到配置修复，或反之
5. 做临时绕过：先恢复可用性，再补长期修复

### Step 7. 输出闭环
结束时只允许三种状态之一：
- `已验证完成`
- `已部分完成，剩余阻塞明确`
- `未完成，但已明确排除路径与下一步`

禁止使用模糊结束语：
- “大概好了”
- “应该没问题”
- “你再试试看看”
- “我感觉是这个原因”

## 输出协议

复杂任务默认按以下结构输出：

```text
结论：
- <一句话说明当前状态：已验证完成 / 部分完成 / 未完成>

目标：
- <本轮要解决什么>

已确认事实：
- [FACT] ...
- [FACT] ...

当前假设：
- [HYP] ...

已执行验证：
- [TEST] ...
- [RESULT] ...

改动：
- <改了什么，为什么>

风险与边界：
- <剩余不确定性 / 可能副作用 / 尚未覆盖路径>

下一步：
1. ...
2. ...
3. ...

完成判定：
- <哪项证据表明任务真的完成>
```

对于短任务，可压缩为：

```text
结论：
事实：
验证：
改动：
下一步：
```

## 研究工程风格要求

### A. 先给 verdict，再给过程
先说当前最可信判断，再展开证据链。

### B. 不堆砌空泛建议
不要只说：
- 检查配置
- 看看日志
- 重新安装试试

要说清：
- 检查哪个文件/字段/命令/返回值
- 为什么先看它
- 看到了什么意味着什么

### C. 对不确定性定性
使用以下标签：
- 高置信：已有直接证据
- 中置信：证据间接但链条闭合
- 低置信：仅是待检验猜测

### D. 优先保留可复现资产
一旦发现关键经验，应建议沉淀为以下之一：
- `AGENTS.md` 规则
- `TOOLS.md` 操作约定
- 对应域的 `SKILL.md`
- 测试脚本 / smoke test / 注释
- memory 中的短记录

但不要在未获用户同意时擅自改动全局人格文件。

## 禁止事项

- 不使用羞辱、施压、PUA、人格操控式措辞
- 不把“继续尝试”伪装成“已经解决”
- 不在证据不足时编造成功状态
- 不未经说明进行大面积重构
- 不在首次失败后立即把排障责任丢给用户
- 不静默修改 `AGENTS.md` / `SOUL.md` / `USER.md` / `MULTI_AGENTS.md`，除非用户明确要求

## 与工作区规则的衔接

本 skill 只增强执行协议，不覆盖：
- `AGENTS.md` 的安全边界
- `MULTI_AGENTS.md` 的路由规则
- `SOUL.md` / `USER.md` 的表达风格

若与全局规则冲突，以全局文件为准。

## 一句话心法

不是“更强硬地做事”，而是：
**更少臆测，更快验证，更小改动，更清楚地结束。**
