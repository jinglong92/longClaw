## 1. 系统总览：你到底在构建什么

你不是在“聊天自动化”，你在构建一个个人决策操作系统（Personal Decision OS）：

- 有明确角色分工（总控 vs 专职）
- 有通信协议（谁什么时候说、说几轮、产出什么）
- 有裁决逻辑（冲突时按规则决策，而非情绪）
- 有记忆系统（历史经验可复用）
- 有反馈回路（指标驱动持续优化）

一句话：**从“问答助手”升级到“决策系统”。**

## 2. 第一性原理：为什么很多多代理系统会失败

### 2.1 常见失败原因
1. 代理太多，边界不清（人人都能回答，等于没人负责）
2. 没有协议（交流自由化，信息污染）
3. 没有裁决器（冲突时互相拉扯）
4. 没有记忆治理（把噪音当经验写入）
5. 没有指标（只凭“感觉不错”）

### 2.2 成功的最小条件
- **唯一责任中心**：Controller 必须有最终裁决权
- **并行受限**：并行是例外，默认单代理
- **输出标准化**：专职统一格式，便于总控聚合
- **记忆分层**：短期临时信息与长期知识隔离
- **可回滚**：坏规则可以撤销

## 3. 核心架构：1个总控 + N个专职 + 规则层 + 记忆层

```text
User
  -> Controller (意图识别/路由/裁决/汇总)
      -> Specialist(s) (单域分析与建议)
          -> Controller (统一输出)
              -> User

并行限制：最多2个专职，且必须满足跨域阻塞触发条件
```

### 3.1 角色定义
- **Controller（CTRL）**：路由、优先级、冲突裁决、最终输出口径
- **Specialists**（如 JOB/LIFE/PARENT/WORK/LEARN/MONEY）：仅本域分析

### 3.2 关键设计原则
- 专职代理不越权
- 总控不做细活（避免角色污染）
- 所有最终建议必须可执行（下一步<=3条）

## 4. SOUL 层：系统人格与价值观

你提到“SOUL”，本质是**行为哲学层**，决定系统在不确定场景里的默认倾向。

### 4.1 SOUL 的作用
当规则未覆盖时，SOUL 决定系统怎么选：
- 保守 or 激进
- 舒适优先 or 成长优先
- 和稀泥 or 直言挑战

### 4.2 你的 SOUL 约束（已明确）
- 坦诚直言，不迎合
- 主动挑战假设
- 指出逻辑漏洞/借口/侥幸
- 建议要有事实、推理、步骤
- 优先长期成长，而非短期舒适

### 4.3 SOUL 不等于“语气”
SOUL 是决策风格，不是文字风格：
- “礼貌”可以保留
- “模糊、讨好、回避判断”必须去掉

## 5. Memory 层：短期、长期、可验证记忆

### 5.1 记忆分层模型

#### A. Working Memory（工作记忆）
- 当前会话临时信息
- 生命周期：分钟到小时
- 例：刚发来的岗位截图细节

#### B. Episodic Memory（事件记忆）
- 某天发生了什么
- 生命周期：天到周
- 例：今天集中投递、看了哪些岗位

#### C. Semantic Memory（长期语义记忆）
- 稳定偏好/长期目标/高复用规则
- 生命周期：月到年
- 例：你偏好“反驳式顾问”风格

### 5.2 长期记忆写入标准（必须同时满足）
1. 稳定性：未来大概率仍有效
2. 决策相关：会影响后续建议
3. 可验证：不是纯情绪噪音

### 5.3 记忆污染防控
- 不把单次情绪写入长期记忆
- 冲突信息要标注时间与来源
- 定期清理失效记忆（至少月度）

### 5.4 记忆读取优先级
1. 用户显式新指令（最高）
2. 长期偏好
3. 近期事件
4. 默认系统策略

## 6. 协议层（Protocol）：让协作可控

协议是“多代理系统的API契约”。

### 6.1 路由协议
- 用户 `@角色`：直达该角色（总控仍汇总）
- 用户未 `@`：总控自动判域 + 公布路由路径

### 6.2 专职输出协议（固定4段）
1. 结论（1句）
2. 行动项（<=3）
3. 风险（<=2）
4. 是否需总控决策（是/否）

### 6.3 跨域协作协议
触发条件：
- 同一任务涉及2域
- 存在互相阻塞

约束：
- 最多2个专职
- 最多2轮
- 必须产出：共识 / 分歧 / 待裁决

### 6.4 变更协议（防漂移）
- 专职不能互改规则
- 所有规则变更需提交 Controller 审批
- 支持回滚（版本化）

## 7. 路由层（Routing）：如何选对代理

### 7.1 路由流程
1. 意图识别（主问题是什么）
2. 领域映射（属于哪个专职）
3. 阻塞判断（是否需第二专职）
4. 输出路由路径（可解释）

### 7.2 你的领域映射（建议）
- JOB：岗位、简历、面试、offer
- LIFE：日程、健康、生活管理
- PARENT：育儿、教育、家庭照护
- WORK：职场策略与晋升
- LEARN：学习路径与能力补齐
- MONEY：预算、投资、风险管理

### 7.3 路由质量指标
- Top-1 路由准确率
- 次优路由后悔率（是否应选别的代理）
- 多代理触发率（越低越稳，非越高越好）

## 8. 冲突裁决层：如何在多目标中做取舍

### 8.1 优先级分层（你当前最实用）
- P0：安全/健康/法律
- P1：硬截止且不可延期
- P2：高价值机会窗口
- P3：生活/家庭刚需
- P4：可延期优化

### 8.2 同级 Tie-breaker
1. 不可逆损失大者优先
2. 不可委托性高者优先
3. 沉没成本高者优先
4. 连锁影响大者优先

### 8.3 裁决输出模板
- 决策：选A不选B（1句话+理由）
- 补救：被延后任务何时补、怎么补
- 是否需你拍板：是/否

## 9. 多代理交流学习机制：学，但不失控

### 9.1 每日异步学习会（10分钟）
每个专职提交3条：
- 今日有效策略
- 今日失败/风险
- 可复用规则

### 9.2 周度知识治理
- Controller 汇总《跨域经验卡》
- 对规则变更做审批/拒绝/回滚

### 9.3 学习质量指标
- 复用率（建议被采纳次数）
- 误导率（导致返工/被否决）
- 业务贡献（是否推动关键目标）

## 10. 偏好管理：把“你是谁”变成系统约束

### 10.1 偏好类型
1. 沟通偏好（直白/简洁/结构化）
2. 决策偏好（成长优先/风险厌恶度）
3. 节奏偏好（低频高质量）

### 10.2 优先级
`用户显式偏好 > 默认系统风格`

### 10.3 你的关键偏好（系统必须遵守）
- 主动挑战你的推理
- 识别借口与侥幸
- 给出明确可执行步骤
- 不给模棱两可答案

---

## 11. 评估与度量：如何判断系统在变好

### 11.1 四类核心指标
1. **正确性**：路由准不准、建议是否可执行
2. **效率**：冲突解决时间、往返轮次
3. **稳定性**：输出一致性、规则漂移率
4. **价值性**：对实际结果的推动（投递、面试、家庭安排）

### 11.2 你的推荐指标看板
- Route Accuracy >= 80%
- Conflict Resolution < 15 min
- Mislead Rate < 10%
- Weekly Reuse Count 持续上升

## 12. 失败模式与回滚机制

### 12.1 典型失败模式
- 代理越权（边界崩溃）
- 并行泛化（每题都多代理）
- 记忆污染（把临时噪音当长期事实）
- 规则膨胀（规则越来越多却没人执行）

### 12.2 回滚触发条件
- 连续2天冲突>3次
- 误导率超过阈值
- 用户反馈“更乱了”

### 12.3 回滚策略
1. 降级到单代理模式
2. 暂停跨域协作
3. 回退到上一个稳定规则版本
4. 只保留Top 5有效规则

## 最后一段（给你）

你真正该追求的，不是“AI替你思考”，而是：

- 你定义规则
- 系统执行规则
- 你审计结果
- 系统持续进化

做到这一步，你就是系统架构师，而不是工具使用者。
