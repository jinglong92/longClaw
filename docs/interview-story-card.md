## 二、三层递进故事（10 分钟版）

### 第一层：OR 基础（3 分钟）

**问题**：日均 500 万订单、20 万换电任务，人工经验调度，成本失控

**我做了什么**：

- DRL 派单策略 → 求解延迟 127ms（基线 4s），46.8% 请求质量改善 1.4%

**结果**：节省近亿现金流，确立行业代差壁垒

**叙事锚点**：
> "这不是研究项目，是全国异步并行部署的生产系统，
> 我对线上 A/B 指标负责。"

### 第二层：AI 增强 OR（4 分钟）

**新问题 A：运维贵**
人工排查换电问题 30min/次，运维团队被淹没在重复性工作里

**解法**：诊断 Agent（LangGraph + ReAct + SFT+GRPO）
```
4 类 LangGraph 诊断工作流
  → 覆盖 55% 高频问题
  → 复杂长尾：ReAct + few-shot，匹配率 31%
  → SFT 格式合规率接近 100%，工具分层准确率 96%
  → GRPO 工具路径匹配率提至 60%+
```

**新问题 B：OR 建模周期长**
每次新业务场景，算法工程师要重新建模、调启发式，周期 2-4 周

**解法**：Solver Agent（EoH 驱动启发式进化搜索，ICML 2024）
```
LLM 驱动启发式函数进化，替代固定启发式
  → 目标函数值改善 16.3%
  → 求解失败率降低 13.3%
```

**叙事锚点**：
> "我理解 AI 不是替代 OR，而是增强 OR 的开发和运维效率。
> 诊断 Agent 解决运维成本，Solver Agent 解决建模周期。"

### 第三层：Agent 系统本身（3 分钟）

**新问题：做 Agent 系统本身很难**
记忆系统怎么设计？多模型怎么协同？幻觉怎么控制？
这些问题在美团内部没有现成答案，我需要自己验证。

**解法**：longClaw 个人 AI 助理（2026.03 — 至今，Mac mini M4 24/7 运行）

**三个核心验证**：

```
验证 1：记忆系统
  三层架构（上下文/每日日志/长期记忆）
  四级 route-aware 检索（同域近期→同域全量→跨域兜底）
  FTS + Hybrid Embedding（RRF fusion）
  结论：按域精准注入节省 ~80% token，文件记忆优于向量数据库

验证 2：多模型协同
  CTRL 总控 + 10 专职代理 + 置信度仲裁（P0-P4）
  Subagent 并发（search/memory/heartbeat agent，最小权限隔离）
  结论：多模型协同需要仲裁协议，不是简单路由

验证 3：执行完整性与幻觉控制
  Harness 层 Deny > Ask > Allow 三层权限
  借鉴 Claude Code 内部架构（sourcemap 逆向分析）
  PostToolUse 强制注入，Anti-stall 规则
  结论：幻觉控制要放在基础设施层，不能靠 LLM 自觉
```

**叙事锚点**：
> "longClaw 是我的实验田，不是玩具。
> 它让我在生产环境里验证了 Agent 系统的核心架构决策，
> 这些决策直接对应 Shopee 这个岗位要做的事。"

## 四、Coding Agent 的串联方式

**为什么 Coding Agent 和这个 JD 相关**：

JD 写了"推动从单轮问答到多轮长程 Agent 的能力演进，包含任务规划、状态追踪与上下文管理"。

Coding Agent 正是这个演进的具体实现：

```
单轮（现状）：
  ENGINEER 专职给建议 → 用户自己去改代码

多轮长程（目标）：
  code-agent skill
    Step 1：repo-explorer 自主探索 codebase（任务规划）
    Step 2：制定修改计划，用户确认
    Step 3：执行修改，每步 readback 验证（状态追踪）
    Step 4：运行测试，失败换路（最多 2 次）
    Step 5：交付可验证的 diff（上下文管理）
```

**面试时可以这样说**：
> "我不只是用过 Coding Agent，我自己实现了一个。
> 核心挑战是 Codebase 理解——LLM 不知道代码在哪里。
> 我设计了 repo-explorer subagent，借鉴 SWE-agent 的 ACI 设计，
> 用结构化文件地图解决这个问题。
> 下一步是加 repo-map（tree-sitter 解析 AST），对应 Aider 的核心技术。"

## 五、高频追问准备

**Q1：你的记忆系统和 RAG 有什么区别？**
> "RAG 是检索增强生成，解决的是'怎么找到相关文档'。
> 我的记忆系统解决的是'怎么在多次对话之间保持连续性'。
> 两者都有检索，但记忆系统还有写入、老化、跨 session 沉淀。
> 具体来说，我用了四级 route-aware 检索——先按路由域收窄范围，
> 再做 FTS + Hybrid Embedding 的 RRF 融合，
> 节省了约 80% 的 token 注入量。"

**Q2：你怎么解决幻觉问题？**
> "我的核心判断是：幻觉控制要放在基础设施层，不能靠 LLM 自觉。
> 具体做了三件事：
> 1. Harness 层 Deny > Ask > Allow 三层权限，Deny 规则在 hook 前生效
> 2. PostToolUse 强制注入——工具调用结果必须写入 DEV LOG，不能靠 LLM 自述
> 3. Anti-stall 规则——`doing:` 只在同轮已发起工具调用时才允许说
> 这套设计借鉴了 Claude Code 的内部架构（sourcemap 逆向分析）。"

**Q3：你对 ADK 了解多少？**
> "我没有直接用过 ADK，但我深度使用了 LangGraph。
> 两者在架构理念上高度一致——都是状态图 + 工具编排 + 条件路由。
> LangGraph 的 StateGraph 对应 ADK 的 Agent Graph，
> LangGraph 的 ToolNode 对应 ADK 的 Tool 接口。
> 我相信上手 ADK 不需要太长时间，核心概念我都有实践基础。"

**Q4：你的 SFT+GRPO 训练效果怎么验证的？**
> "两个维度：
> 1. 离线指标：格式合规率（SFT 后接近 100%），工具路径匹配率（GRPO 后 60%+）
> 2. 在线验证：通过预留回放评测流程做 baseline vs candidate 对比
> 我设计了三层 Dense Reward：格式（0-0.2）/ 工具路径（0-0.4）/ 证据引用（0-0.4），
> 这个设计直接来自美团换电诊断 Agent 的训练经验。"

**Q5：多模型协同最难的地方是什么？**
> "不是路由，是仲裁。
> 路由解决的是'派谁去'，仲裁解决的是'多个专家给出不同答案时怎么办'。
> 我设计了置信度协议：每个专家输出必须附置信度和依据，
> ≥0.8 直接采纳，0.6-0.8 标注不确定性，<0.6 追问。
> 冲突时显式展示分歧，CTRL 给倾向性建议。
> 这比简单的'谁置信度高用谁'更可解释，也更容易调试。"
