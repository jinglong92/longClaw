# OpenClaw 面试讲解提纲（2026-03-25）

## 一句话定位
OpenClaw 是把 LLM 从“会聊天”升级为“可执行、可审计、可路由”的 Agent Runtime。

## 设计目标（Why）
1. 真实任务执行稳定化，而非 Demo 对话
2. 路由可控（谁处理）
3. 安全可控（边界/权限/回退）
4. 过程可追踪（日志/审计/复盘）
5. 跨会话连续性（记忆层）

## 核心架构（What）
- **CTRL Orchestrator**：意图识别、路由决策、冲突仲裁、统一对外交付
- **Specialists**：按域拆分（JOB/WORK/LIFE/ENGINEER…）
- **Tools**：文件、执行、检索、会话编排等一等公民能力
- **Memory**：长期记忆（MEMORY）+ 日记记忆（memory/YYYY-MM-DD）
- **Audit/Replay**：请求级日志、路由可见、失败复盘

## 关键取舍（How）
1. **默认单专职，必要时并行双专职**：降低复杂度与 token 成本
2. **CTRL 唯一对外交付**：避免多代理口径冲突
3. **风险审计前置**：高影响任务优先可控性
4. **工具优先于纯文本幻想执行**：保证可验证与可复现

## 面试高频追问与回答框架
### Q1: 为什么不直接用一个通用 Agent？
- 单 Agent 在复杂任务里容易上下文漂移与职责耦合
- 专职拆分 + CTRL 仲裁能提高稳定性与可解释性

### Q2: 什么时候并行？
- 仅在跨域任务或明显盲区时并行
- 平时单代理优先，避免并发冲突和成本上升

### Q3: 如何防止幻觉/错误传播？
- 工具参数 schema 校验
- 超时与重试策略
- 失败降级（fallback / human-in-loop）
- 全链路 trace 支持定位与回放

## 工程价值（Result）
- 可追踪：知道“为什么这么答”
- 可复现：同输入可回放
- 可降级：失败可控，不是全或无
- 可优化：用成功率、时延、成本、回退率持续迭代

## 个人贡献讲法（结合 AI+OR 背景）
“我专长是把 Agent Runtime 用在优化决策场景：
LLM 负责策略生成，Solver 保证可行与质量，Evaluator 负责风险闭环，最终做成可上线系统，而不是提示词 Demo。”

## 2 分钟口述模板
1. 先说目标：可执行 + 可审计 + 可路由
2. 再说架构：CTRL + Specialists + Tools + Memory + Audit
3. 再说取舍：单代理默认、并行克制、风险前置
4. 最后说结果：稳定性/成本/时延/回退率
